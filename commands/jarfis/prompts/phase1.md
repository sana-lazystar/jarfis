**Step 1-2.5: PRD Completeness Check — Ratchet** (executed directly by orchestrator — not an agent)

> This step is executed directly by the orchestrator without invoking an agent.
> Before the Gate 1 user confirmation, it automatically checks PRD quality to prevent ambiguous PRDs from propagating to subsequent Phases.
> **Ratchet rule**: Once an item has Passed (2 points), it must remain Passed even after PO rewrites.

Verification checklist (5 items, 0-2 points each):
1. **Ambiguous expressions** — Detect terms like "appropriate/fast/sufficient/as needed/to be decided later" → 0: 3+, 1: 1-2, 2: all specific numbers
2. **KPI measurability** — Each KPI has a number (target value) + measurement method → 0: none/qualitative, 1: numbers only, 2: numbers + measurement method
3. **Performance Budget specificity** — Each metric has specific numbers + measurement method → 0: undefined, 1: partial, 2: all defined
4. **Required Roles rationale** — Each role's rationale field has 1+ sentences → 0: none, 1: partial, 2: all present
5. **Scope boundaries** — "Out of Scope" section → 0: undefined, 1: inclusions only, 2: inclusions + exclusions

Verification result handling:
- **All 5 items Pass (2 points)**: Proceed directly to Gate 1
- **Fail items exist**: Instruct the PO (senior-product-owner) to revise the PRD, specifying only the Fail items
  - **Ratchet verification**: After rewrite, if a previously Passed item changes to Fail, this is a ratchet violation → re-instruct PO to "improve only Fail items while maintaining quality of passed items"
  - Repeat up to 2 times. If Fail items still exist afterward, warn the user at Gate 1 with "PRD Score: X/10, Fail: [items]"

**Rules to include when instructing PO rewrite** (do not share the scoring criteria themselves):
- "Do not degrade the quality of previously passed items"
- "Focus improvements only on items flagged as Fail"
- "Reinforce only the relevant sections while maintaining the existing PRD structure"
